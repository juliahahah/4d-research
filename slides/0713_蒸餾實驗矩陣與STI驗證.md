# 研究討論整理(2026/07/13)— 知識蒸餾實驗矩陣 + STI 驗證

## 老師的核心指示

不要用「一套蒸餾方法套所有模態」的粗暴做法;針對不同模態配對「比較適合它的蒸餾方式」。
**目標:找出「哪個知識(模態)該用哪種蒸餾」。**
定位:這組實驗是「哪個模態值得帶進 RL reward」的前期篩選,結果餵給後續 RL 設計。

## 步驟 2:做實驗——用 4D-RGPT 的 code,直接在它的消融上加實驗

**Code**:[github.com/NVlabs/4D-RGPT](https://github.com/NVlabs/4D-RGPT)(已確認公開)。只降低訓練步數壓成本。
**Base 模型**:第一階段用 **NVILA-Lite-8B** 復刻(蒸餾訓練綁死 NVILA);第二階段移植 **Qwen2.5-VL**(目前只有 eval 支援)。移植為第二階段工作。
**復刻目標**:先把它的 ablation 補齊/復刻——STI 接近 35(zero-shot 基線 33.8)。
**Eval**:用 STI 完整驗證(很快);先部分資料驗證想法,不一次全上。

### 實驗兩軸

**軸一:Knowledge(蒸什麼)**
矩陣範圍 = L4P 現成訊號:**depth / flow(optical)/ motion / camray**;設單一模態 = 逐模態消融。
逐格實驗為逐模態獨立蒸(原論文四模態同時蒸、λ 加權),變因固定。
**不在範圍**:Scene flow / 3D trace / RigidFlow / DINO——L4P 不吐這些訊號,自接 teacher 為大工程,不選 teacher。

**軸二:Distillation Method(怎麼蒸)**
三種方法(另兩種待學長確認後補):
- **Logit / response KD**:對齊最終輸出分布(soft label)→ 通常跟分類模型有關。code = `--ed_weights`(L_ED per-task decoder,現成)。
- **Feature KD**:對齊中間層特徵圖 → 4D-RGPT 用的這種(LD)。code = `--ld_weight`(L_LD,現成)。
- **Relation KD**:對齊樣本/模態間關係(RKD、CRD),含跨模態關聯蒸餾(下式)。repo 沒有、需自寫,列**第二階段**實作並當原創主打;第一階段矩陣先用現成 feature/logit 把軸一鋪滿。
  `L2( cos_sim(teacher_P_depth, teacher_P_flow), cos_sim(student_P_depth, student_P_flow) )`
  物理根據:motion parallax,光流 ∝ 1/深度。實作前要補:①維度對齊(depth 1ch vs flow 2ch)②標量太稀→改逐 patch 相似度圖 ③先做 depth-flow 一對。

### 對應到 code(repo=NVlabs/4D-RGPT)
矩陣兩軸直接對應 `scripts/nvila/sft.sh` 的旗標,不用改架構就能跑:
- **軸一(蒸什麼)** = `--distill_tasks`,子集 of `depth,flow_2d_backward,dyn_mask(motion),camray`;設單一模態 = 逐模態消融。乾淨基線 = `--disable_distillation True`。
- **軸二(怎麼蒸)**:Feature KD = `--ld_weight`(現成)、Logit/Explicit KD = `--ed_weights`(現成)、Relation KD = 自寫。

### 成本確認順序(先便宜、後估算)
「能不能到 35」和「一格要跑多久」是兩件事:
1. **zero-shot NVILA-Lite-8B → STI**:目標 ≈ 33.8,**驗證 eval pipeline 對**(純推論,不用 token)。
2. **官方 4D-RGPT-8B checkpoint → STI**:確認打得到 ~35+,**驗證模型+eval 都對**(gated,要 HF token;仍不付訓練成本)。
3. **全模態短訓練跑一次**:量 wall-clock,× 格數 = 矩陣總 GPU-hr → 決定要不要再砍步數。
- 矩陣**先跑 flow 列**(訊號最強、teacher 現成、直通 RL 主線)。
